# External Integrations

**Analysis Date:** 2026-03-31

## APIs & External Services

**Reverse Geocoding:**
- OpenStreetMap Nominatim (`https://nominatim.openstreetmap.org/search`)
  - Used to resolve ZIP codes found in GPS address strings into human-readable city names
  - Called in `src/gpsParser.js` → `geocodeCityNominatim()`
  - HTTP client: Node.js built-in `https` module (no SDK)
  - Auth: None — public API, no key required
  - Rate limiting: 1.1-second forced delay between requests (`setTimeout(r, 1100)`) to comply with Nominatim's 1 req/sec policy
  - User-Agent header sent: `GeneratorEwidencji/4.0 (contact@example.com)`
  - Result cached in memory (`onlineCitiesCache`) and persisted to `known_cities.json` on disk

## Data Storage

**Databases:**
- None

**File Storage:**
- No persistent cloud storage
- Input files are received as in-memory `Buffer` via `multer` and never written to disk
- Output `.xlsx` files are streamed directly in the HTTP response (`res.send(xlsBuf)`)
- Optional local file: `known_cities.json` written to `process.cwd()` by `src/gpsParser.js` to cache ZIP-to-city lookups across requests; committed to the repo's working directory at runtime

**Caching:**
- In-process JS `Map` (`onlineCitiesCache`) loaded at module initialisation from `known_cities.json` and updated on new Nominatim lookups
- Two hardcoded seed entries in `src/gpsParser.js`:
  - `'43-245'` → `'Studzionka'`
  - `'43-246'` → `'Strumień'`

## Data Formats

**Input:**
- `.xls` — BIFF8/OLE Compound Document format (legacy GPS tracker export). Parsed by hand-written binary parser at `src/xlsParser.js`.
- `.xlsx` — Open XML spreadsheet. Parsed via `exceljs` in `src/gpsParser.js`.
- Both formats accepted by the `/generate` POST endpoint; format determined by file extension.

**Output:**
- `.xlsx` — Open XML spreadsheet generated by `exceljs` in `src/excelGenerator.js`. Returned as a binary response with `Content-Type: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` and `Content-Disposition: attachment`.

**Internal data exchange:**
- All file content travels as Node.js `Buffer` objects between `server.js`, `gpsParser.js`, and `excelGenerator.js`
- No JSON serialisation between modules; plain JS objects and `Map`/`Set` structures are passed directly

## Authentication & Identity

**Auth Provider:** None — the application has no authentication layer. All endpoints are publicly accessible.

## Monitoring & Observability

**Error Tracking:** None — errors are caught in `server.js` and returned as JSON `{ error, detail }` with HTTP 500. `console.error` is used for server-side logging.

**Logs:** `console.log` / `console.error` only; no structured logging library.

## CI/CD & Deployment

**Hosting:**
- Docker container (via `Dockerfile`): targets any container runtime (Cloud Run, etc.)
- `Procfile` (`web: node server.js`) also present for Heroku-style platforms and Firebase App Hosting

**CI Pipeline:** None detected

**Port:** `process.env.PORT` (default `8080`); server binds to `0.0.0.0`

## Webhooks & Callbacks

**Incoming:** None — single `POST /generate` endpoint driven by browser `fetch()`

**Outgoing:**
- Nominatim HTTPS GET requests originating from `src/gpsParser.js` during Excel generation

## Environment Configuration

**Required env vars:** None strictly required — the server starts without any env vars set.

**Optional env vars:**
- `PORT` — TCP port to listen on (defaults to `8080`)

**Secrets location:** None — no API keys, credentials, or secrets are used.
